## M10 — 加载与崩溃（P0）

### BUG-03: 字体首次加载闪烁

**问题**：首次进入游戏时，打字机动画中文字首次出现的瞬间字体不对，动画结束后自动纠正。第二次刷新页面后正常。疑似字体未缓存完成就开始渲染。

**影响**：直接影响新用户的第一印象和信任度。

**修复方案**：
- 使用 `font-display: block` 或在 CSS `@font-face` 中设置，确保字体加载完成前不显示文字
- 或者在应用初始化时使用 `document.fonts.ready` 等待字体加载完成后再开始渲染
- 预加载关键字体：在 `index.html` 中添加 `<link rel="preload">` 标签

**验收标准**：首次访问时所有文字从一开始就是正确字体，无闪烁或替换。

---

### FEAT-08: 加载页（Loading Screen）

**问题**：游戏缺少加载缓冲，素材（字体、图片）未加载完就开始渲染，导致首屏体验差。

**修改方案**：

1. **新增加载页**
   - 使用已有的海报图片作为加载背景：`03-src/frontend/public/images/poster-titled.png`
   - 页面中央显示游戏标题
   - 底部显示加载进度条（字体 + 图片加载进度）
   - 加载完成后显示 **"Start Game"** 按钮，由用户主动点击进入

2. **加载逻辑**
   - 预加载所有字体文件（`document.fonts.ready`）
   - 预加载所有NPC立绘和背景图（`new Image().onload`）
   - 预加载对话JSON数据
   - 全部就绪后激活 "Start Game" 按钮（从灰色变亮）

3. **视觉设计**
   - 海报背景 + 半透明暗色遮罩
   - 标题使用游戏主字体
   - 进度条采用古风样式（羊皮纸/蜡封风格）
   - "Start Game" 按钮采用蜡封印章风格（与v3的合约确认按钮一致）

**验收标准**：
- 首次访问先看到加载页，素材全部加载完毕后才可进入游戏
- 后续访问可利用缓存快速跳过（但保留短暂品牌展示）

### FEAT-09: Day5市场崩盘表现优化

**问题**：Day5市场崩盘时的闪烁效果粗糙（"low"），不符合游戏基调。

**修改方案**：

1. **移除闪烁效果**，改为静态但有冲击力的表现：
   - 价格看板上的价格数字用红色加删除线显示原价，旁边显示暴跌后的新价
   - 顶部显示一条横幅：**"今日几乎无人签署买入合约"**
   - 或显示：**"市场冻结 —— 无人报价"**

2. **Day5仍然可以操作**
   - 玩家仍然可以尝试卖出（但价格极低）
   - 可以买入VOC合约和房产契约（作为"避险"选择）
   - 大部分玩家会直接点击"查看结局"，但保留操作自由度
   - 这也为v4的隐藏彩蛋（FEAT-11）提供操作空间

**验收标准**：Day5不再有闪烁效果，用文字传达崩盘氛围，且不影响操作。

### BUG-04: Day5无法买入VOC合约

**问题**：最后一天（Day5）无法买入VOC合约，疑似交易引擎在崩盘日错误地禁用了所有买入操作。

**修复方案**：
- 检查 `tradingEngine.ts` 中Day5的交易逻辑
- 确保"崩盘日禁用买入"只针对郁金香品种，不影响房产契约和VOC合约
- VOC和房产在Day5应仍可正常交易（价格小幅下跌，但功能不受限）

**验收标准**：Day5可以正常买入/卖出VOC合约和房产契约。

### FEAT-10: 人物担忧动画优化

**问题**：
1. 担忧状态下的人物持续跳动很烦人（annoying）
2. Day4所有NPC都进入"担忧"状态不合理——应该有人仍然狂热

**修改方案**：

1. **跳动动画限制**
   - 忧虑动画最多跳3下，然后停止
   - 或改为轻微的上下浮动（很小的幅度），而非剧烈跳动
   - 保持忧虑表情/图标，去掉持续跳动

2. **Day4情绪分布调整**

   | NPC | Day4情绪 | 理由 |
   |---|---|---|
   | Hendrik | 狂热/激动 | 赌徒，越涨越兴奋，卖房梭哈的人不可能担忧 |
   | Cornelis | 微微忧虑 | 老花商，开始感觉到不对劲 |
   | Anna | 忧虑 | 谨慎寡妇，一直在担心 |
   | Maria | 冷静/看透 | 老板娘，始终清醒 |
   | Stranger | 神秘/不表露 | 不动声色 |

**验收标准**：担忧动画不再持续跳动；Day4各NPC情绪符合角色性格。

### FEAT-11: 卖出"全部"按钮显示具体数量

**问题**：卖出时的"全部"按钮只显示"全部"文字，用户不知道具体卖出多少。

**修改方案**：
- "全部"按钮改为显示实际数量：**"全部 (7)"** 或 **"全部 (12份)"**
- 数量随持仓实时更新
- 买入时同样可以显示"最大可购 (5)"（基于当前资金）

**验收标准**：买卖操作中，数量相关按钮都显示具体数值。

### FEAT-12: 域名交易（元游戏彩蛋）🃏

**问题**：一个有趣的idea——既然游戏主题是"郁金香泡沫"，为什么不把这个游戏的域名也拿来"交易"？

**概念**：
- 在游戏结算页面末尾或历史板块中，展示一个"交易此域名"的入口
- 用户可以登录后报价，"买入"这个游戏本身
- 接入302支付，价高者得
- 形成"交易中的交易"——游戏主题的完美闭环

**技术方案（需进一步评估）**：

| 方案 | 描述 | 优劣 |
|---|---|---|
| A. 纯报价模式 | 用户只报价不付款，联系域主线下交易 | 简单，但交易可能流产 |
| B. 托管支付 | 302支付托管，类似竞拍 | 需要处理税费、退款、争议 |
| C. NFT化 | 铸造域名为NFT，链上交易 | 技术复杂，但最有话题性 |

**待解决问题**：
- 收款中的税费处理
- 退款机制（出价后被更高价超越时）
- 域名过户的法律流程
- 是否需要"当前最高出价"实时显示

**优先级说明**：标记为P0是因为这是营销亮点，但技术实现可以在M10-M12期间并行准备，不必阻塞其他Bug修复。实际可降级到独立分支开发。

**验收标准**：至少实现方案A（报价登记），有入口和展示页。

## M11 — 结算与彩蛋（P1）

### FEAT-13: 啤酒购买记录

**问题**：与老板娘对话买啤酒时，扣除费用但未在交易历史中留下记录。对话选项也未标注价格。

**修改方案**：
1. 对话选项中啤酒购买项添加价格标注：
   - 当前：`"来杯啤酒吧"`
   - 修改为：`"来杯啤酒吧（花费 0.1 荷兰盾）"`

2. 啤酒购买写入交易历史：
   - 类型标记为"消费"（区别于"合约交易"）
   - 记录：日期、商品名、数量、单价、总价
   - 在交易历史面板中单独分组成"酒馆消费"类

**验收标准**：啤酒购买有价格提示，且出现在交易历史中。

### FEAT-14: 隐藏结局彩蛋 —— "神来之笔"

**问题**：如果玩家在最后一天卖掉所有郁金香并买入房产或VOC，这是最聪明的操作，应当得到特殊奖励。

**触发条件**（同时满足）：
- Day5结束时，持有0份郁金香合约
- Day5结束时，持有 > 0 份房产契约 或 VOC合约

**彩蛋内容**：

> **隐藏结局：神来之笔** ✨
>
> 你不仅全身而退，而且明智地转向了长期投资。
>
> 随着后来荷兰经济的崛起，你的资产还在继续翻倍。
>
> VOC的股票在随后的几十年里稳步增长，运河房产的价值也水涨船高。
> 而那些执迷于郁金香的人，只能在酒馆里追忆往昔。
>
> *作者甚至想投胎当你的后人。*

**展示方式**：
- 结局页面显示特殊的金色边框
- "神来之笔"用金色字体突出
- 可配合烟花/金粉等庆祝动画（简短）

**验收标准**：满足条件时显示隐藏结局，不满足时不出现。

---

### FEAT-15: 结局文案细化 —— 基于持仓状态

**问题**：当前结局只看总财富，不区分玩家是"持有郁金香"还是"已清仓"，缺少对玩家决策的针对性评价。

**修改方案**：在结局展示前，根据持仓状态追加不同的提示语：

| 持仓状态 | 总财富 | 追加提示 |
|---|---|---|
| 仍持有郁金香 且 盈利 | > 初始资金 | "你虽然在这几天资产翻倍，但后来郁金香的价格越来越低，你最终所剩无几了。" |
| 仍持有郁金香 且 亏损 | ≤ 初始资金 | "你在这场操作中一败涂地，即便最后也不认输。后来郁金香的价格越来越低，你最终赔得分文不值。" |
| 已清仓郁金香（且不满足彩蛋条件） | — | 保持原有结局逻辑不变 |

**与彩蛋的优先级**：先检测FEAT-14的彩蛋条件，不满足彩蛋再检测此规则。

**验收标准**：结局文案反映玩家的持仓决策，不只是看总财富数字。
